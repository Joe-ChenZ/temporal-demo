# temporal_simple_agent — Coding Agent with a Hand-Written Loop

The same coding agent as `temporal_coding_agent/`, but with **no agent framework**. You
write the loop. It's built on plain Temporal (activities + a workflow) plus a single
LiteLLM call — the pattern from Temporal's
[LiteLLM cookbook](https://docs.temporal.io/), extended from one call into an agent loop.

The point: this is where you actually *see* the agent flow. The whole loop is ~30 lines in
`workflows.py`.

---

## Files

```
temporal_simple_agent/
├── models.py       # pydantic AssistantMessage / ToolCall — the typed message shape
├── activities.py   # llm_call (LiteLLM, returns AssistantMessage) + read/write/run tools
├── workflows.py    # SimpleCodingAgent — the hand-written agent loop + tool schemas
├── worker.py       # plain Temporal worker (no plugin) + pydantic_data_converter
├── starter.py      # kicks off one task (same data converter)
└── workspace/      # created on first run; the agent's sandboxed working dir
```

**Typing.** `llm_call` validates LiteLLM's raw reply into a pydantic `AssistantMessage`
(`models.py`), and both clients use `pydantic_data_converter` so that typed object crosses
the activity boundary intact. The loop codes against attributes — `message.tool_calls`,
`tc.function.name`, `tc.id` — instead of guessing dict keys, and a malformed reply fails
loudly at the boundary instead of deep in the loop.

---

## The loop (the whole agent)

```
messages = [system, user task]
loop (up to 20×):
    message = llm_call(model, messages, TOOLS)     # one Claude call, as an activity
    append message to history
    if no tool_calls:  return message.content      # model is done
    for each tool_call:
        result = run the matching tool activity    # read_file / write_file / run_command
        append {role: tool, tool_call_id, result}  # feed the result back
```

The model decides *which* tool, *with what args*, and *when to stop* — that model-driven
control flow is what makes it an agent. Each `llm_call` and each tool is a Temporal
activity, so the loop is durable: crash the worker, restart, completed steps are replayed,
not re-run.

This agent runs one prompt and exits. To make it multi-turn/conversational (a long-running
session), see [`NOTES.md`](NOTES.md).

## Setup & run

Same as the other demos. Needs `ANTHROPIC_API_KEY` in the repo-root `.env`, a running
`temporal server start-dev`, and `uv sync`.

```bash
# Terminal 1
uv run python temporal_simple_agent/worker.py
# Terminal 2
uv run python temporal_simple_agent/starter.py "Write fib.py with a fib(n) function, run fib(10), print the result"
```

Files land in `temporal_simple_agent/workspace/`. Watch the Web UI (http://localhost:8233):
you'll see alternating `llm_call` and tool activities — that's the loop, one round trip per
activity.

## vs `temporal_coding_agent` (the plugin version)

Same behavior, same tools, same model. The difference is who writes the loop:

| | `temporal_coding_agent` (plugin) | `temporal_simple_agent` (this) |
|---|---|---|
| Agent loop | `Runner.run` (OpenAI Agents SDK) — hidden | hand-written `for` loop in the workflow |
| Tool schemas | auto-generated by `activity_as_tool` | hand-written `TOOLS` list |
| Tool dispatch | `activity_as_tool` | explicit `if name == ...` |
| Deps | `temporalio[openai-agents,opentelemetry]` + litellm | just litellm (+ plain temporalio) |
| You learn | how to *configure* an agent | how an agent actually *works* |

Both reach Claude through LiteLLM; both are durable. This one is the "build it yourself"
version — read `workflows.py` top to bottom and you've seen the entire agent.
